2. HARDWARE SPECIFICATIONS
================================================================================

MAIN PROCESSOR:
- Raspberry Pi RP2040 microcontroller
- Dual-core ARM Cortex-M0+ @ 133MHz
- 264KB SRAM, 2MB on-chip flash
- 30x GPIO pins (multi-function)
- Operating voltage: 1.8V to 5.5V (core: 1.1V)

SENSORS:
1. LSM6DS3TR-C IMU (6-axis gyroscope + accelerometer)
   - Interface: SPI1
   - Chip Select: GPIO9
   - Shared SPI bus: SCK=GPIO10, MOSI=GPIO11, MISO=GPIO8
   - Gyro range: ±125 to ±2000 dps
   - Accel range: ±2 to ±16 g
   - Max sample rate: 6.66 kHz
   - Device ID: 0x6A (verified in test firmware)

2. DPS310XTSA1 Barometer (pressure + temperature)
   - Interface: SPI1 (shared with IMU)
   - Chip Select: GPIO12
   - Pressure range: 300-1200 hPa
   - Accuracy: ±0.002 hPa (±0.02m altitude)
   - Temperature: -40°C to +85°C
   - Device ID: 0x10 (verified in test firmware)

3. GPS Module (external via UART1)
   - Interface: UART1 (GPIO4=TX, GPIO5=RX)
   - Baud rate: 9600
   - Connector: 6-pin JST-GH
   - Compatible: Ublox NEO-M8N, M9N
   - Magnetometer: HMC5883L/QMC5883L on I2C0

STORAGE:
- W25Q128JVSIQ SPI Flash (128Mbit / 16MB)
  - Interface: QSPI
  - Speed: up to 104MHz
  - Usage: Firmware, parameters, logs

- MicroSD Card Slot (503398-1892)
  - Interface: SPI0
  - CS=GPIO17, MOSI=GPIO19, MISO=GPIO16, SCK=GPIO18
  - Filesystem: FAT32 (not exFAT)

OUTPUTS:
1. RGB LED (FM-B2020RGBA-HG)
   - Red: GPIO13 (PWM6_B) - Active LOW
   - Green: GPIO14 (PWM7_A) - Active LOW
   - Blue: GPIO15 (PWM7_B) - Active LOW
   - CRITICAL: Writing 0 = LED ON, Writing 1 = LED OFF
   - Startup sequence confirmed working: Red → Green → Blue

2. Buzzer (QMB-09B-03 passive buzzer)
   - Pin: GPIO25 (PWM4_B)
   - Type: Passive (requires PWM frequency)
   - Resonant frequency: 2700Hz ±300Hz (2400-3000Hz range)
   - Active HIGH (PWM enabled = sound)
   - Current implementation: Police siren (cycles 2400/2700/3000Hz @ 200ms for 5s)

3. ESC Outputs (4x motor control)
   - ESC0: GPIO20 (PWM2_A) - Motor 1 (Front Right)
   - ESC1: GPIO21 (PWM2_B) - Motor 2 (Rear Right)
   - ESC2: GPIO22 (PWM3_A) - Motor 3 (Rear Left)
   - ESC3: GPIO23 (PWM3_B) - Motor 4 (Front Left)
   - PWM: 50Hz standard or 400Hz Oneshot125
   - Pulse width: 1000-2000µs
   - Connector: 3-pin JST-GH each (GND, +5V, Signal)

INPUTS:
1. RC Input (Radio Control receiver)
   - Pin: GPIO24
   - Protocols: PPM/SBUS/DSM
   - Connector: 3-pin JST-GH
   - WARNING: SBUS is inverted serial (may need level shifter)

2. Arming Button
   - Pin: GPIO26 (ADC0)
   - Circuit: Button to ground, internal pull-up
   - Released: 3.3V (ADC=4095), Pressed: 0V (ADC=0)

3. Battery Monitoring
   - Voltage: GPIO27 (ADC1)
   - Current: GPIO28 (ADC2)
   - USB Detect: GPIO29 (ADC3) - monitors VBUS

   CRITICAL SAFETY ISSUE - BATTERY VOLTAGE DIVIDER:
   Current firmware config: BOARD_BATTERY_V_DIV = 1.74
   This implies max safe input: 3.3V / 1.74 = 5.74V
   
   For 4S LiPo battery:
   - Nominal: 14.8V (3.7V × 4 cells)
   - Fully charged: 16.8V (4.2V × 4 cells)
   
   PROBLEM: 16.8V at this divider would result in 16.8V × 1.74 = 29.23V at ADC!
   This WILL DAMAGE the RP2040 (ADC max = 3.3V)
   
   REQUIRED ACTION: Verify voltage divider resistors from schematic
   Minimum divider ratio: 16.8V / 3.3V = 5.09
   
   From BOM: R3=27.4Ω, R4=27.4Ω (need to verify actual circuit topology)

COMMUNICATION INTERFACES:
1. UART0 - Telemetry/Console
   - TX: GPIO0
   - RX: GPIO1
   - Baud: 115200 (8N1)
   - Connector: 6-pin JST-GH
   - Pinout:
     Pin 1: GND
     Pin 2: TX (GPIO0)
     Pin 3: RX (GPIO1)
     Pin 4: SDA/CTS (GPIO2) - optional
     Pin 5: SCL/RTS (GPIO3) - optional
     Pin 6: +5V
   - Current use: Serial console for NuttX shell
   - Future use: MAVLink telemetry

2. USB
   - Type: Micro-USB
   - Functions: Firmware flashing (UF2 bootloader), USB CDC-ACM console
   - USB Console Status: UNRELIABLE (garbled output)
   - Root cause: NuttX TinyUSB driver timing issues on RP2040
   - Solution: Use UART console instead

3. SWD Debug Port
   - SWCLK: Debug clock
   - SWDIO: Debug data
   - Connector: 3-pin header (GND, SWCLK, SWDIO)
   - Debugger: Picoprobe (OpenOCD + GDB)

POWER:
- Input: USB 5V or Battery 7.4V-16.8V
- 3.3V Regulator: SSP1117-3.3V LDO (1A max)
- Internal 1.1V regulator for RP2040 core (DVDD)
- Decoupling: 15× 100nF, 2× 1µF, 3× 10µF capacitors

Power Tree:
Battery (7.4V-16.8V) → 5V Regulator → SSP1117 3.3V LDO
                                    ↓
                          RP2040 + Sensors + Peripherals
USB VBUS (5V) ──────────────────┘

3. FIRMWARE ARCHITECTURES

TWO DISTINCT FIRMWARE TYPES:

A. PX4 AUTOPILOT FIRMWARE (Production target)
   Location: firmware/openfc2040/rsp_2040/
   Upstream: firmware/openfc2040/px4-autopilot/ (submodule)

   - NuttX RTOS (real-time operating system)
   - PX4 flight stack (attitude estimation, controllers)
   - MAVLink telemetry protocol
   - uORB inter-process communication
   - NSH shell (NuttShell console)

   Output: raspberrypi_pico_minimal.uf2 (1.1 MB)
   Console: UART0 (115200 baud) via GPIO0/GPIO1
   
   Build process:
   1. CMake reads 100+ CMakeLists.txt files
   2. Ninja compiles ~500 source files in parallel
   3. Linker combines all objects into .elf
   4. objcopy converts to .bin
   5. uf2conv.py converts to .uf2 (RP2040 bootloader format)
   Total time: 5-15 minutes first build, 30s incremental
   
   Flash size: 52.41% (1.1MB / 2MB)
   RAM usage: 6.44% (17KB / 264KB)

B. PERIPHERAL TEST FIRMWARE (Hardware validation)
   Location: firmware/test-firmware/peripherals_testing/
   SDK: Pico SDK (submodule at pico-sdk/)
   
   Purpose: Simple standalone firmware to test:
   - IMU sensor communication (SPI)
   - Barometer sensor communication (SPI)
   - GPS UART communication
   - RGB LED control (PWM)
   - Buzzer control (PWM)
   - ESC PWM outputs
   - USB CDC console
   
   Build: CMake + Ninja
   Output: openfc2040_firmware.uf2 (86.5 KB)
   Console: USB CDC-ACM (/dev/ttyACM0)
   
   Commands available:
   - help: Show all commands
   - imu: Read accelerometer + gyroscope
   - baro: Read pressure + temperature
   - gps: Read GPS NMEA data
   - sensors: Read device IDs (IMU=0x6A, Baro=0x10)
   - red/green/blue/off: LED control
   - beep: Buzzer test (plays police siren)
   - live: Live sensor data stream
   - calib: Calibration mode
   - debug: Debug info
   
   Key implementations:
   - Software bit-bang SPI (no hardware SPI peripheral used)
   - PWM buzzer with dynamic frequency control
   - GPS on UART1 (GPIO4/5) - CRITICAL FIX: moved from UART0
   - LED startup sequence: Red → Green → Blue (confirms active-low)

PX4 BUILD PROCESS:
```bash
cd firmware/openfc2040/rsp_2040
make raspberrypi_pico_minimal
# or
./scripts/build.sh
```

Step-by-step what happens:
1. CMake Configuration Phase
   - Reads platforms/nuttx/CMakeLists.txt
   - Reads boards/raspberrypi/pico/CMakeLists.txt
   - Finds all .cpp/.c source files
   - Generates build.ninja (20,000+ lines)

2. Compilation Phase (Ninja)
   - Compiles ~500 files to .o objects
   - Uses arm-none-eabi-gcc with flags:
     -mcpu=cortex-m0plus
     -mthumb
     -O2
     -g
   - Parallel compilation (uses all CPU cores)

3. Linking Phase
   - arm-none-eabi-ld combines all .o files
   - Links against NuttX RTOS library
   - Resolves function calls between modules
   - Output: raspberrypi_pico_minimal.elf (~5MB with debug)

4. Binary Generation
   - objcopy converts .elf to .bin
   - uf2conv.py wraps .bin with UF2 header
   - Final: raspberrypi_pico_minimal.uf2 (1.1MB)

COMMON BUILD ISSUES AND SOLUTIONS:

1. CMake cache error "different directory than where created"
   Symptom: Error about CMakeCache.txt path mismatch
   Cause: Project moved to different directory
   Solution: rm -rf build && ./scripts/build.sh

2. Submodule not initialized
   Symptom: "pico-sdk not found" or "px4-autopilot empty"
   Solution: git submodule update --init --recursive

3. ARM toolchain not found
   Symptom: "arm-none-eabi-gcc: not found"
   Solution: sudo apt install gcc-arm-none-eabi

4. Permission denied on USB device
   Symptom: Cannot access /dev/ttyACM0 or /dev/ttyUSB0
   Solution: sudo usermod -a -G dialout $USER (logout/login required)

ISSUE #1: USB CONSOLE GARBLED OUTPUT (RESOLVED - USE UART INSTEAD)
Date: October 2025
Symptom: USB CDC-ACM console (/dev/ttyACM0) showing random characters
Tried:
- Different baud rates (9600, 57600, 115200)
- CONFIG_DEV_CONSOLE enabled/disabled in NuttX
- Different USB cables
- Different terminal programs (picocom, screen, minicom)
Root cause: NuttX TinyUSB driver has timing issues on RP2040
Solution: Switched to UART console on GPIO0/GPIO1 (UART0)
Result: Clean console output at 115200 baud
Documentation: firmware/openfc2040/rsp_2040/USB_CONSOLE_INVESTIGATION.md
Status: RESOLVED - UART console is industry standard anyway

ISSUE #2: PERIPHERAL TEST FIRMWARE BLANK CONSOLE (RESOLVED)
Date: November 4, 2025
Symptom: Flashed test firmware, /dev/ttyACM0 detected but empty output
Debug steps:
1. Verified LED startup sequence visible (Red→Green→Blue) - firmware running
2. Checked USB CDC initialization in main.c - present
3. Tested different terminals (picocom, screen) - same result
4. Reviewed stdio configuration - stdio_init_all() called
5. Checked UART conflict - FOUND ISSUE

Root cause: GPS initialization code was calling uart_init(uart0, 9600)
This reconfigured UART0 pins (GPIO0/GPIO1) which are used by stdio_init_all()
for USB CDC console on RP2040.

Code before fix:
```c
#define GPS_UART uart0  // WRONG - conflicts with USB stdio
#define GPS_TX_PIN 0    // WRONG - conflicts with console
#define GPS_RX_PIN 1    // WRONG - conflicts with console

void gps_init(void) {
    uart_init(GPS_UART, GPS_BAUD_RATE);  // Breaks stdio!
    gpio_set_function(GPS_TX_PIN, GPIO_FUNC_UART);
    gpio_set_function(GPS_RX_PIN, GPIO_FUNC_UART);
}
```

Code after fix:
```c
#define GPS_UART uart1  // Moved to UART1
#define GPS_TX_PIN 4    // GPIO4 (UART1_TX)
#define GPS_RX_PIN 5    // GPIO5 (UART1_RX)

void gps_init(void) {
    uart_init(GPS_UART, GPS_BAUD_RATE);  // Now uses UART1
    gpio_set_function(GPS_TX_PIN, GPIO_FUNC_UART);
    gpio_set_function(GPS_RX_PIN, GPIO_FUNC_UART);
}
```

Added comment in code:
```c
// NOTE: GPS uses UART1 (GPIO4/GPIO5), NOT UART0 (GPIO0/GPIO1)
// UART0 is reserved for USB console (stdio)
```

Solution: Moved GPS to UART1 (GPIO4/GPIO5)
Result: Console working, GPS can communicate on separate UART
Files modified: firmware/test-firmware/peripherals_testing/src/main.c
Status: RESOLVED

ISSUE #3: BUZZER SIMPLE GPIO VS PWM (RESOLVED)
Date: November 4, 2025
Symptom: Buzzer not making sound, only clicks
Initial implementation: gpio_put(BUZZER_PIN, 1) - DC signal
Problem: QMB-09B-03 is PASSIVE buzzer, needs frequency tone
Solution: Implemented PWM-based control

Implementation:
```c
void buzzer_init(void) {
    gpio_set_function(BUZZER_PIN, GPIO_FUNC_PWM);
    uint slice_num = pwm_gpio_to_slice_num(BUZZER_PIN);
    
    // For 2700Hz: 125MHz / (50 * 926) = 2700.2 Hz
    pwm_set_clkdiv(slice_num, 50.0f);
    pwm_set_wrap(slice_num, 925);
    pwm_set_gpio_level(BUZZER_PIN, 463);  // 50% duty cycle
    pwm_set_enabled(slice_num, false);  // Start off
}

void buzzer_set_frequency(uint16_t frequency_hz) {
    uint slice_num = pwm_gpio_to_slice_num(BUZZER_PIN);
    uint16_t wrap = (125000000 / (50 * frequency_hz)) - 1;
    pwm_set_wrap(slice_num, wrap);
    pwm_set_gpio_level(BUZZER_PIN, wrap / 2);
}
```

Police siren implementation:
```c
void buzzer_startup_sequence(void) {
    uint slice_num = pwm_gpio_to_slice_num(BUZZER_PIN);
    pwm_set_enabled(slice_num, true);
    
    absolute_time_t start_time = get_absolute_time();
    uint16_t frequencies[] = {2400, 2700, 3000};
    uint8_t freq_index = 0;
    
    // Run for 5 seconds
    while (absolute_time_diff_us(start_time, get_absolute_time()) < 5000000) {
        buzzer_set_frequency(frequencies[freq_index]);
        sleep_ms(200);  // Change every 200ms
        freq_index = (freq_index + 1) % 3;
    }
    
    pwm_set_enabled(slice_num, false);
}
```

Result: Working buzzer with police siren startup (2400-3000Hz cycling)
Status: RESOLVED

ISSUE #5: GIT COLLABORATION WORKFLOW QUESTION (RESOLVED)
Date: November 4, 2025
Question: "Do I need to push changes or will they auto-sync?"
Answer: Git requires MANUAL operations:
- Changes are LOCAL until you git push
- Remote changes don't auto-download (need git pull)
- Workflow:
  1. git pull (fetch remote changes before working)
  2. Make changes, git add, git commit
  3. git push (upload to GitHub)
- If push fails with "rejected" error:
  git pull --rebase (merge remote changes)
  git push (try again)

11. LESSONS LEARNED AND BEST PRACTICES
================================================================================

UART PIN CONFLICTS:
- RP2040 stdio_init_all() uses UART0 for USB CDC console
- Never use uart_init(uart0) for peripherals when USB console enabled
- Always use separate UART (uart1) for GPS/telemetry when stdio is active
- Document UART usage clearly in code comments

CMAKE BUILD CACHE:
- CMake caches absolute paths in build/ directory
- Moving project location requires: rm -rf build && rebuild
- Always use clean build after path changes
- Git ignore build/ directories

PASSIVE VS ACTIVE BUZZERS:
- Passive buzzers need PWM frequency (2-4kHz typical)
- Active buzzers just need DC voltage (gpio_put)
- QMB-09B-03 is PASSIVE, resonant at 2700Hz
- Use hardware PWM for clean tones, not software toggling

LED POLARITY:
- Always check schematic for active-high vs active-low
- OpenFC2040 RGB LED is active-LOW (write 0 to turn on)
- Test with simple blink before complex patterns
- Document polarity in code comments

GIT COLLABORATION:
- Always git pull before starting work
- Changes don't auto-sync (manual push required)
- Use git pull --rebase to handle remote conflicts
- Commit frequently with clear messages
- Update NEXT_STEPS.md after every session

DOCUMENTATION:
- Keep documentation IN SYNC with code changes
- Document WHY decisions were made, not just WHAT
- Include error messages and solutions in docs
- Mark CRITICAL issues prominently
- Update task tracker after every work session

SAFETY:
- NEVER connect LiPo battery without verifying voltage divider
- NEVER test ESC outputs with propellers attached
- Use current-limited power supply for initial testing
- Double-check GPIO pin assignments before powering on

DEBUGGING:
- Start with simplest test (LED blink)
- Use printf debugging via serial console
- Check device IDs before reading sensor data
- Verify SPI communication before complex transactions
- LED patterns useful when console not available

BUILD SYSTEM:
- Understand full build pipeline (CMake → Ninja → Linker → UF2)
- Clean rebuild if strange linking errors
- Check submodule status if missing files
- Verify toolchain versions if compilation fails
