### Root Cause Analysis

**Error Pattern Identified**:
```
PD1TASpL1245lEr!=+     (FD A: tries read 1245, gets error errno=4/5)
PD1TBSpL1245lEr!=+     (FD B: tries read 1245, gets error errno=4/5)
PD1TCSpL1245lEr!=+     (FD C: tries read 1245, gets error errno=4/5)

```

**Marker Interpretation**:
- `1` = read() call attempt
- `2` = read() returned (zero or error)
- `4` = error code detected
- `5` = transient error (EAGAIN/EBADF/ENOTTY/EIO)
- `r` = errno value being examined
- `E` = EOF/error path in nsh_session
- `r` = retry attempt

**Cause Chain**:
1. **Boot console redirected to RAM**: defconfig comment indicates boot output diverted to buffer, not actual UART
2. **Inherited FDs non-functional**: FD 0/1/2 are write-only (stdout/stderr to RAM) or invalid
3. **rebind_stdio() fails completely**:
   - FD 0 check → fcntl fails or flags=O_WRONLY
   - FD 1 (stdout) → try nsh_rp2040_fd_usable_input() → "optimistically" true but read() fails
   - FD 2 (stderr) → same failure
   - Stream-based rebind → all streams non-readable
   - Inherited FD probing (3-255) → all return EBADF/ENOTTY/EIO on read()
4. **Cannot safely open new FD**: Calling `open("/dev/ttyS0", O_RDWR)` during NSH brings **immediate SMP deadlock** (inode lock contention in VFS layer)

### What Was Attempted

Code changes made and results:
1. **O_NONBLOCK clearing** (nsh_session.c): Calls `fcntl(infd, F_SETFL, flags & ~O_NONBLOCK)` → no change, FDs still error on read()
2. **Lenient FD checking** (nsh_consolemain.c): Made `nsh_rp2040_fd_can_read()` accept fcntl failure → no improvement
3. **readline retry loop** (readline_fd.c): Added sleep+retry on EAGAIN → infinite loop accumulation
4. **Explicit open fallback** (nsh_consolemain.c): Tried `open("/dev/ttyS0", O_RDWR)` → IMMEDIATE HANG/DEADLOCK confirmed
5. **Immediate EOF on errors** (readline_fd.c): Return EOF on first EAGAIN/EBADF → NSH retries FDs but all fail same way

The earlier documented "NSH visible prompt" (`PD1TSpL18a...`) can now be understood:
- System reached NSH, printed prompts successfully
- But those were **single-shot tests without input**
- Once prompt loop tried to read from stdin, infinite EAGAIN cycle began
- The "degradation to `18ldoCMmnop245...`" was memory corruption from **infinite retry looping**, not actual progress

**SMP Deadlock Risk**:
- Opening any device during NSH runtime → VFS inode lock contested with other threads

### `uart_dev_s` Field Offset Calculation

From `include/nuttx/serial/serial.h`, the struct contains:

```c
struct uart_buffer_s {
  sem_t            sem;    /* 4 bytes  (int16_t semcount + uint16_t flags) */
  volatile int16_t head;   /* 2 bytes */
  volatile int16_t tail;   /* 2 bytes */
  int16_t          size;   /* 2 bytes */
  /* 2 bytes implicit padding */
  char            *buffer; /* 4 bytes (pointer) */
};  // total = 16 bytes ... but sizeof = 24 due to sem alignment
```

C program (`gcc -m32`-equivalent for Cortex-M0+ 32-bit) confirms:
```

sizeof(uart_buffer_s) = 24   (0x18)
sizeof(uart_dev_s)    = 136  (0x88)

offsetof(recv)        = 64   (0x40)   from g_uart0port base
offsetof(recv.buffer) = 80   (0x50)   from g_uart0port base
offsetof(ops)         = 88   (0x58)   from g_uart0port base
```

With `g_uart0port @ 0x200000a0`:
- `recv.buffer` lives at SRAM `0x200000f0`
- `ops`         lives at SRAM `0x200000f8`

These are exactly the two fields documented as being corrupted (`dev->recv.buffer → 0x7829F627`, `dev->ops->rxint → crash`).

---

### Canary Guard System (Already in Place)

The `rp2040_serial.c` already implements a full self-repair mechanism via `rp2040_uart_diag_repair()`:
- Guards: `RP2040_UART_CANARY_PRE = 0x13579BDF`, `RP2040_UART_CANARY_POST = 0x2468ACE0`
- Canaries on both sides of `g_uart0port`, `g_uart0priv`, `g_uart0rxguard`
- If any field in `g_uart0port` is corrupted (incl. `recv.buffer`, `recv.size`, `ops`), this function detects and restores them.
- The `rx.buffer` is restored to `g_uart0rxguard.data` (inside the 512-byte guarded ring buffer at `0x20000354`).
- Marker `C` in runtime trace = "corruption bypass/repair engaged"; `M` = "bypass complete".

**The repair works but corruption recurs** — so the write source is active and asynchronous (not a one-time init bug).

---

### Root Cause: `CONFIG_IDLETHREAD_STACKSIZE=750` — Critically Undersized for SMP

Audit of all stack sizes in `boards/raspberrypi/pico/nuttx-config/nsh/defconfig`:

```
CONFIG_ARCH_INTERRUPTSTACK=4096      (OK — 4 KB for IRQ stack per CPU)
CONFIG_IDLETHREAD_STACKSIZE=750      ← SMOKING GUN: 750 BYTES ONLY
CONFIG_INIT_STACKSIZE=3194           (OK for NSH main task)
CONFIG_SCHED_HPWORKSTACKSIZE=1280    (borderline)
CONFIG_SCHED_LPWORKSTACKSIZE=1632    (OK)
CONFIG_STACK_COLORATION=y            (stack paint enabled — good)
```

**Problem**: With `CONFIG_SMP=y`, both Core 0 AND Core 1 run idle threads that share this stack size. The Core 1 idle thread must:
1. Execute `arm_smp_cpu_started()` — not trivial
2. Run `nx_idle_trampoline()` through NuttX SMP init path
3. Enter the actual idle loop which calls work queue and scheduler hooks

750 bytes (~188 32-bit words) is insufficient for this chain on Cortex-M0+. A typical SMP idle bringup needs ~1.5–2 KB before it settles into the lightweight idle loop.

**What happens when it overflows**: The Core 1 idle stack grows **downward** from its allocated top. When it overflows past its base/bottom, it writes to **whatever SRAM lies below the allocation**. Given `g_uart0port` is near `0x200000a0` (very low in SRAM), and the idle TCB stack is dynamically allocated by NuttX's `arm_createstack()`, if the allocation lands above this region, an overflow reaches it.

**Why corruption is periodic / recurs**: The idle thread runs repeatedly (whenever a CPU is idle). Each time through the idle loop the stack is partially re-traversed, causing repeated corruption. The canary repair fixes it per-call but isn't called on every idle wakeup.

---

### Fix Applied (2026-04-01)

**`boards/raspberrypi/pico/nuttx-config/nsh/defconfig`**:
```diff
-CONFIG_IDLETHREAD_STACKSIZE=750

```

### If Corruption Persists After Stack Fix

If `BAD...` markers still appear after rebuilding with `IDLETHREAD_STACKSIZE=2048`, the next suspects are:
1. **`CONFIG_SCHED_HPWORKSTACKSIZE=1280`** — the high-priority work queue may also overflow under SMP load; try bumping to 2048.
2. **ISR writing out-of-bounds** — if `rxbuf->head` wraps with wrong modulo (e.g., due to `int16_t` sign extension on Cortex-M0+), the write `rxbuf->buffer[head] = ch` could land at an arbitrary address. Instrument `uart_recvchars()` to assert `head < size` before writing.
3. **Core 1 interrupt stack** — `CONFIG_ARCH_INTERRUPTSTACK=4096` is 4 KB shared; if both cores share one interrupt stack region and Core 1's ISR nesting is deep, it could overflow from the top of `g_intstack_alloc` downward toward lower SRAM. Check `rp2040_cpuidlestack.c` for per-CPU stack setup.
